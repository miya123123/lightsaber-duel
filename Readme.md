# Lightsaber Duel（ライトセイバー・デュエル）

ライトセイバーを構えたプレイヤーが、アリーナでAIファイターと1対1で戦うブラウザゲームです。移動、旋回、ジャンプ、ダッシュを組み合わせ、ライトセイバーを相手の身体へ接触させて勝利を目指します。

## デモ

[GitHub Pagesでプレイする](https://miya123123.github.io/lightsaber-duel/)

## ゲームの特徴

- **接触型のライトセイバー戦闘**: 攻撃ボタンを連打する方式ではなく、キャラクターの向きと位置を操作してセーバーを相手へ当てます。
- **自由なトップダウン移動**: 前後左右だけでなく、斜め方向にも移動できます。
- **手動旋回**: プレイヤーの向きを`Q`と`F`で調整し、セーバーの狙いを定められます。
- **ジャンプとダッシュ**: ジャンプで相手を飛び越え、ダッシュで一気に間合いを詰めたり離したりできます。
- **状況に応じて動くAI**: AIはプレイヤーへゆっくり向き直り、距離に応じて接近、回避、旋回、ジャンプ、ダッシュを使います。
- **セーバー同士の衝突**: セーバーがぶつかると、両者が反対方向へ弾かれます。
- **レスポンシブ表示**: デスクトップとモバイル幅の両方で、アリーナとHUDが画面内に収まります。

## 遊び方

### 操作方法

| 操作 | キー | 内容 |
| --- | --- | --- |
| 移動 | `W` `A` `S` `D` または矢印キー | 上下左右・斜め方向へ移動 |
| 左旋回 | `Q` | プレイヤーとセーバーを反時計回りに旋回 |
| 右旋回 | `F` | プレイヤーとセーバーを時計回りに旋回 |
| ジャンプ | `E` | 向いている方向、または入力中の移動方向へ跳ぶ |
| ダッシュ | `Space` | 向いている方向、または入力中の移動方向へ高速移動 |
| リセット | `R` | 対戦を最初からやり直す |
| 再戦 | 勝敗決着後の「再戦」ボタン | 新しい対戦を開始 |

> 現在の操作はキーボード向けです。画面レイアウトはモバイル幅に対応していますが、タッチ操作UIは実装されていません。

### 勝利条件

1. 移動と旋回で、ライトセイバーの先を相手の身体へ当てます。
2. 命中すると相手の体力が減り、ノックバックが発生します。
3. AIの体力を先に`0`にするとプレイヤーの勝利です。

ジャンプ中は双方の身体への攻撃判定とセーバー同士の衝突が無効になります。また、ジャンプとダッシュにはクールダウンがあり、同時には使用できません。画面上部の体力バーと状態表示を見ながら間合いを調整してください。

## 使用したスキル

ここでいう「スキル」は、Codexが各工程を一定の手順と品質基準で進めるために参照した専門ワークフローです。ゲームに組み込まれるライブラリや実行プログラムそのものではありません。

| スキル | 担当した工程 | 本プロジェクトでの具体的な役割 |
| --- | --- | --- |
| **Game Studio** | 企画・技術選定 | 1対1のブラウザゲームという要件を整理し、2D描画、入力、当たり判定、AI、HUD、テストをどのように構成するか決めるために使用しました。 |
| **Phaser 2D Game** | ゲーム実装 | PhaserのSceneを中心に、ゲームループ、キーボード入力、キャラクター描画、アニメーション、HUD連携を実装する指針として使用しました。描画を`DuelScene`、対戦ルールを`DuelSystem`へ分ける構成にも反映されています。 |
| **Game Playtest** | プレイ品質確認 | 実際に対戦を操作し、移動、旋回、ジャンプ、ダッシュ、命中、セーバー衝突、勝敗表示を確認しました。HUDがプレイ領域を妨げないか、モバイル幅で表示が崩れないかも確認対象にしました。 |
| **Playwright Interactive** | 対話的ブラウザ検証 | Chromiumを起動した状態でキー入力を送り、ページ上に公開したゲーム状態を読み取り、スクリーンショットとコンソールを確認しました。自動テストだけでは把握しにくい操作感や表示も検証しました。 |
| **Image Gen / imagegen** | 画像原画生成 | SFアリーナ、プレイヤー、AIファイター、ジャンプ動作用シートの原画を生成しました。生成物は`public/assets/generated/`へ保存しています。 |
| **agent-sprite-forge（generate2dsprite）** | 2D画像加工 | 生成したジャンプシートから背景を透明化し、4フレームへの分割、位置と大きさの整列、透過PNGシートの再構成、確認用GIFとメタデータの出力、端切れなどの品質確認を行いました。 |
| **OpenAI Docs** | 仕様確認 | 画像生成を行う際に、利用するOpenAI画像生成機能の入出力や指定方法を公式情報で確認するために使用しました。 |
| **develop-web-game** | 継続的な開発・再検証 | 小さな変更ごとにゲームを起動し、短い入力操作、状態取得、スクリーンショット、コンソールエラー確認を繰り返す開発ループに使用しました。README作成時にも、記載した操作と現行実装が一致するか再確認しました。 |

複数のスキルは独立しているのではなく、**Game Studioで方針を定め、Phaser 2D Gameで実装し、Image Genとgenerate2dspriteで画像を用意し、Game Playtest・Playwright Interactive・develop-web-gameで検証する**という流れで連携しています。

## 開発ツール

「開発ツール」は、コードの作成、画像生成、ビルド、実行、テストで実際に利用したモデル、ライブラリ、言語、コマンドライン環境です。

| ツール | 種別 | 本プロジェクトでの用途 |
| --- | --- | --- |
| **Codex（GPT-5.5・高）** | AI開発エージェント | 当時の開発セッションで、要件整理、設計、TypeScript実装、画像生成工程の進行、テスト実行、問題修正、README作成までを一貫して担当しました。モデル名と推論設定は、その開発時に使用した構成を示しています。 |
| **image_gen** | 画像生成ツール | テキストで指定した構図や配色から、アリーナとキャラクターのラスター画像を生成しました。生成後の透過処理やフレーム分割は別工程で行っています。 |
| **Phaser 3** | 2Dゲームフレームワーク | Canvas/WebGL描画、Sceneのライフサイクル、画像アセットの読み込み、キーボード入力、アニメーション、画面サイズへの追従を担います。`DuelScene.ts`が主な利用箇所です。 |
| **TypeScript** | 実装言語 | プレイヤーとAIの状態、移動、クールダウン、AI判断、幾何計算、命中判定、ノックバック、勝敗判定を型付きコードで実装しました。ビルド時には`tsc`で型の整合性も検査します。 |
| **Vite** | 開発サーバー・ビルドツール | 開発中のローカルサーバーを起動し、TypeScriptとPhaserをブラウザで実行できる形へまとめます。本番ビルドではGitHub Pagesへ配置する静的ファイルを`dist/`に生成します。 |
| **Node.js / npm** | JavaScript実行環境・パッケージ管理 | Phaser、Vite、TypeScript、Playwrightの依存関係を`package-lock.json`に基づいて導入し、`npm run dev`、`npm run build`、`npm test`などの共通コマンドを実行します。 |
| **Playwright** | E2Eテストツール | Chromium上で実際のページを開き、キー入力とゲーム状態を検証します。`playwright.config.ts`ではデスクトップChrome相当の`1440 x 900`とPixel 7相当の`412 x 915`を対象にしています。失敗時はトレースとスクリーンショットを保持します。 |

開発時は、`npm run dev`でViteを起動してCodexとブラウザ検証スキルが操作し、完成確認では`npm run build`によるTypeScript・本番バンドル検証と、`npm test`によるPlaywright E2Eテストを実行する構成です。

## セットアップ

```bash
npm ci
```

Playwrightのブラウザが未導入の場合は、次のコマンドも実行します。

```bash
npx playwright install chromium
```

## 起動方法

開発サーバーを起動します。

```bash
npm run dev
```

ターミナルに表示されたURLをブラウザで開いてください。既定のURLは`http://127.0.0.1:5173`です。終了するときは、起動したターミナルで`Ctrl+C`を押します。

## ビルドとテスト

### 本番ビルド

```bash
npm run build
```

TypeScriptの型チェック後、配布用ファイルが`dist/`へ生成されます。

### 自動テスト

```bash
npm test
```

本番ビルド後にPlaywrightテストを実行し、次のような振る舞いをデスクトップとモバイル幅で確認します。

- 初期表示、HUD、操作ガイド
- プレイヤーとAIの移動・旋回・ジャンプ・ダッシュ
- ダッシュのクールダウン
- ジャンプ中の回避と長距離移動
- セーバーの身体命中判定とノックバック
- セーバー同士の衝突
- モバイル幅でのCanvasとHUDの表示

ビルド結果をローカルで確認する場合は、次のコマンドを使用します。

```bash
npm run preview
```

## 実装メモ

- ゲーム内部の論理解像度は`1280 x 720`です。Phaserの`FIT`スケーリングで画面サイズへ合わせます。
- 対戦ルールは描画から分離され、`DuelSystem`が位置、速度、体力、AI、当たり判定、勝敗を管理します。
- キャラクター画像は`public/assets/generated/`から読み込み、アリーナはPhaserの図形描画で構成します。
- ライセンスは現時点で明記されていません。再配布・商用利用を検討する場合は、コードと画像素材の権利条件を別途確認してください。

## 今後の改修事項

### 優先度: 高

- **プレイヤー画像の透明化品質を修正する**: `player_topdown_cutout.png`の輪郭、ライトセイバーの発光周辺、身体や腕の隙間に残る不要な不透明部分・背景色・縁取りを除去します。透明化後は暗色、明色、チェック柄の各背景へ重ね、残留ピクセルや白・黒のハローがないことを確認します。
- **ジャンプ画像も同じ基準で再処理する**: `player_jump_sheet.png`と`player_jump_frames/`の全4フレームを個別に検査し、透過漏れ、半透明のごみ、輪郭欠けを修正します。修正後にシート、個別PNG、確認用GIF、メタデータを同じ生成元から更新します。
- **待機・ジャンプ間の表示ずれを抑える**: 通常立ち絵とジャンプシートで異なる画像サイズ、原点、縮尺を整理し、アニメーション切り替え時にキャラクターやライトセイバーが跳ねて見えないよう統一します。
- **見た目と当たり判定を再調整する**: 画像の透明化・位置合わせ後、描画されたライトセイバーと`DuelSystem`のセーバー線分が一致しているか確認し、身体命中とセーバー衝突の判定位置を再調整します。
- **サウンドを実装・洗練する**: BGM、ライトセイバーの駆動音、セーバー衝突、身体命中、ジャンプ、ダッシュ、勝利・敗北の効果音を追加します。`DuelEvent`やキャラクター状態と再生を連動させ、同じ音の過剰な連続再生を抑えながら、音量調整、ミュート、ブラウザの自動再生制限に対応します。
- **戦闘エフェクトを実装・洗練する**: セーバーの発光と軌跡、衝突時の火花、命中時のフラッシュ・ヒットストップ・画面振動、ダッシュ残像、ジャンプ着地、勝敗演出を強化します。各現象を見分けられる色・形・表示時間に整理し、モバイル端末でも視認性とフレームレートを維持できるよう粒子数や描画負荷を調整します。

### 優先度: 中

- **タッチ操作を追加する**: 現在はモバイル幅で表示できますが操作はキーボード専用です。移動、旋回、ジャンプ、ダッシュ、リセット用のタッチUIを追加し、プレイ領域を隠さない配置にします。
- **ブラウザテストを安定化する**: 実時間のキー入力時間に依存する移動テストを、決定的な時間進行または状態到達待ちへ変更し、端末性能による数ピクセルの差で失敗しないようにします。
- **画像アセットの自動QCを強化する**: 全フレームを対象に、端への接触、孤立ピクセル、アルファ境界、フレーム間の重心・縮尺差を検査し、再生成時の品質低下を検出できるようにします。

### 優先度: 低

- **初期バンドルを軽量化する**: 現在の本番ビルドではJavaScriptチャンクが500kBを超える警告が出ます。Phaserやゲームコードの分割、不要アセットの読み込み確認を行い、初回表示までの転送量を削減します。
- **アセット権利情報を整備する**: 生成画像、加工画像、コードの利用条件をまとめ、再配布・公開時に確認できるライセンス表記を追加します。
